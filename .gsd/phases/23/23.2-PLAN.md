# Plan 23.2: Refactorizar Firebase Security Rules (Multi-tenancy)

## Objective
Asegurar que todas las peticiones desde el cliente solo accedan a los datos de la propia organización modificando las Reglas de Seguridad en Firebase (aislamiento estricto).

## Context
- .gsd/ROADMAP.md
- firestore.rules

## Tasks

<task type="auto">
  <name>Actualizar firestore.rules para aislamiento de OrganizationId</name>
  <files>
    - firestore.rules
  </files>
  <action>
    Reescribir `firestore.rules` completo.
    - Quitar la regla insegura `allow read, write: if request.auth != null;`.
    - Crear una función auxiliar: `function getUserOrg() { return get(/databases/$(database)/documents/users/$(request.auth.uid)).data.organizationId; }`
    - Para las colecciones multi-tenant comunes (`crm_contacts`, `inventory`, `sales`, `projects`, `finances`, etc.), requerir `request.auth != null` y obligar a que `resource.data.organizationId == getUserOrg()` o en caso de escritura (create), que envié el mismo `organizationId` (`request.resource.data.organizationId == getUserOrg()`).
    - Permitir que un usuario lea y modifique su propio documento en `users`.
  </action>
  <verify>cat firestore.rules</verify>
  <done>Las reglas de firestore bloquean lectura/escritura cruzada entre organizaciones.</done>
</task>

## Success Criteria
- [ ] El archivo `firestore.rules` incorpora funciones para obtener el `organizationId` del JWT/usuario actual.
- [ ] Los recursos sólo pueden ser leídos y modificados si pertenecen a la misma organización.
